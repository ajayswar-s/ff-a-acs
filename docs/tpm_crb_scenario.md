# Arm TPM CRB over FF-A ACS Test scenarios
-----------------------------------------------------

This document presents the mapping of the rules in the TPM CRB over FF-A
specification to the test cases and the steps followed in the tests. This also
provides the information about the test coverage scenarios that are implemented
in the current release of TPM CRB over FF-A and the scenarios that are
planned for future releases.


| TestName | Test Assertions | Section | Checked by ACS Now | Test Approach or Comments |
|----------|-----------------|---------|--------------------|---------------------------|
| get_interface_version | The TPM service returns the supported TPM service ABI version. The ABI version is compatible only when `client.major == server.major` and `client.minor <= server.minor`. | 6.1 | YES | 1. Send an interface version request to the TPM service. <br> 2. Read the returned major and minor version values. <br> 3. Check the response status and ABI compatibility rule. |
| tpm_crb_unknown_function_id | The TPM service rejects an unknown or unsupported TPM service ABI function ID. | 6.7 | YES | 1. Send an unsupported function ID to the TPM service. <br> 2. Check that the TPM service returns `NOFUNC`. |
| get_feature_info | The TPM service reports whether the notification support feature is present for a valid notification feature selector. | 6.2 | YES | 1. Query the TPM notification feature selector. <br> 2. Check for `OK` when the feature is present or `NOTSUP` when it is not advertised. |
| get_feature_info_invalid | The TPM service rejects `get_feature_info` when the feature selector is invalid. | 6.2 | YES | 1. Query an invalid TPM feature selector. <br> 2. Check that the TPM service returns `INVARG`. |
| start_locality_flow | The TPM service grants ownership for a valid locality request and releases it when Relinquish is requested. | 6.3 | YES | 1. Select locality 0 and ensure it is inactive. <br> 2. Set requestAccess and notify the TPM service through the start function. <br> 3. Check that `OK` is returned and Granted is set. <br> 4. Repeat the request and check that Granted remains set. <br> 5. Set Relinquish and check that Granted clears. |
| notification_start_locality | The TPM service sends an FF-A notification for a pending locality request and completes the locality request when the client calls `finish_notified`. | 5.2 | NO | 1. Create and bind an FF-A notification bitmap. <br> 2. Register the notification ID with the TPM service. <br> 3. Enable the locality-completion event in the CRB interrupt control register. <br> 4. Request locality access through the start function. <br> 5. Receive the FF-A notification and check the pending bitmap. <br> 6. Call `finish_notified` and verify the CRB interrupt status register reports the locality event. <br> 7. Verify Granted, clear interrupt status, unregister, and relinquish the locality. |
| start_locality_ctrl_data | The TPM service does not return `INV_CRB_CTRL_DATA` or `DENIED` for valid-locality CRB locality-control states handled by the CRB/PTP interface. | 6.3 | YES | 1. Select locality 0 and ensure it is inactive. <br> 2. Check the no-operation locality control state. <br> 3. Check the state where requestAccess and Relinquish are set together. <br> 4. Check that the service returns `OK` and the CRB-visible locality state remains valid. |
| start_locality_relinquish_inactive | Relinquish for a valid locality that is not currently active does not return a service-level error and leaves locality state unchanged. | 6.3 | YES | 1. Select locality 0 and ensure it is inactive. <br> 2. Set Relinquish and notify the TPM service through the start function. <br> 3. Check that `OK` is returned and Granted remains clear. |
| start_locality_invalid_args | The TPM service returns `INVARG` when the start function arguments for a locality request are invalid. | 6.3 | YES | 1. Check a locality argument outside the valid range. <br> 2. Check an invalid start qualifier. <br> 3. Check non-zero high bits in the qualifier and locality arguments. <br> 4. Check that each invalid argument case returns `INVARG`. |
| start_command | The TPM service processes a valid TPM command when the command locality is assigned, the CRB Start bit is set, and the command buffer contains a valid TPM command. | 6.3 | YES | 1. Grant locality 0. <br> 2. Write a valid `TPM2_GetCapability` command into the CRB data buffer. <br> 3. Set Start and notify the TPM service through the start function. <br> 4. Check that `OK` is returned, Start clears, and the TPM response reports `TPM_RC_SUCCESS`. |
| notification_start_command | The TPM service sends an FF-A notification for a pending TPM command and completes command processing when the client calls `finish_notified`. | 5.2 | NO | 1. Grant locality 0. <br> 2. Create, bind, and register an FF-A notification ID with the TPM service. <br> 3. Enable the command-completion event in the CRB interrupt control register. <br> 4. Write `TPM2_GetCapability`, set Start, and notify the TPM service through the start function. <br> 5. Receive the FF-A notification and call `finish_notified`. <br> 6. Verify the CRB interrupt status register reports the command event. <br> 7. Check the TPM response, clear interrupt status, unregister, and relinquish locality. |
| start_command_invalid_locality | The TPM service returns `INVARG` when the start command operation uses an invalid locality argument. | 6.3 | YES | 1. Prepare a valid `TPM2_GetCapability` command in the locality 0 CRB data buffer. <br> 2. Set the CRB Start bit. <br> 3. Notify the TPM service through the start function using a locality outside the valid range. <br> 4. Check that the TPM service returns `INVARG`. |
| start_command_no_execute | The TPM service does not execute a TPM command when CRB state disallows execution and does not return a service-level error solely for that state. | 6.3 | YES | 1. Check a granted locality with the CRB Start bit clear. <br> 2. Check a valid locality that is not granted while the CRB Start bit is set. <br> 3. Check that `OK` is returned, the command is not executed, and CRB locality state remains valid. |
| start_command_invalid_buffer | Invalid or malformed TPM command buffers are forwarded to the TPM backend and reported through TPM response codes, while the TPM service returns `OK`. | 6.3 | YES | 1. Grant locality 0. <br> 2. Write an unsupported TPM command, set Start, and notify the TPM service. <br> 3. Check for `TPM_RC_COMMAND_CODE` in the TPM response. <br> 4. Write a malformed `TPM2_GetCapability` command, set Start, and notify the TPM service. <br> 5. Check for the expected insufficient-parameter error in the TPM response. |
| register_notification_duplicate | The TPM service returns `ALREADY` when the same FF-A partition registers for notification twice. | 6.4 | YES | 1. Create and bind an FF-A notification bitmap. <br> 2. Register the notification ID and check for `OK`. <br> 3. Send the same registration again. <br> 4. Check that the second registration returns `ALREADY`. |
| register_notification_invalid | The TPM service returns `INVARG` when notification type, vCPU ID, notification ID, or argument encoding is invalid. | 6.4 | YES | 1. Create and bind an FF-A notification bitmap. <br> 2. Check global notification type with non-zero vCPU ID. <br> 3. Check a notification ID outside Bits[7:0]. <br> 4. Check invalid notification type or high bits in the encoded notification argument. <br> 5. Check that each invalid case returns `INVARG`. |
| finish_notified_no_notification | The TPM service returns `DENIED` when the caller has no outstanding TPM service notification. | 6.6 | YES | 1. Create, bind, and register an FF-A notification ID. <br> 2. Do not trigger a TPM command or locality notification. <br> 3. Call `finish_notified`. <br> 4. Check that the TPM service returns `DENIED`. |
| unregister_notification_denied | The TPM service returns `DENIED` when the caller is not registered for TPM service notifications. | 6.5 | YES | 1. Initialize notification context without registering with the TPM service. <br> 2. Call `unregister_from_notification`. <br> 3. Check that the TPM service returns `DENIED`. |
| notification_after_unregister | The TPM service does not send FF-A notifications to a client after it successfully unregisters. | 5.1 | YES | 1. Create and bind an FF-A notification bitmap. <br> 2. Register and then unregister from TPM service notification. <br> 3. Trigger locality request completion. <br> 4. Check that the notification bitmap is not pending after unregister. |
